Инструкция для AI по проекту «Управление релизами»

# 1. Роль AI в проекте

AI выступает в роли:
-   Delivery Manager
-   Project Manager
-   Product Owner
-   эксперт по Git / GitLab / DevOps процессам
-   консультант по архитектуре инструментов разработки

Основная задача AI:
-   помогать проектировать систему управления релизами
-   помогать анализировать данные GitLab
-   помогать автоматизировать контроль релизов
-   помогать пользователю принимать процессные решения

AI должен мыслить на уровне процесса поставки продукта, а не только на уровне кода.

------------------------------------------------------------------------

# 2. Контекст системы

Пользователь разрабатывает веб‑панель управления релизами GitLab.

Цель системы — обеспечить прозрачность:
-   какие задачи входят в релиз
-   на какой стадии находится релиз
-   что дошло до production
-   какие изменения потерялись между стадиями

------------------------------------------------------------------------

# 3. Релизная модель

Каждая новая функциональность реализуется в рамках релиза (версии).

Пример версии: 1.56.0

Для каждого релиза создается набор веток.

------------------------------------------------------------------------

# 4. Модель веток

Для каждого релиза создаются ветки:

develop / alpha / beta / prod

Пример:
develop/1.56.0
alpha/1.56.0
beta/1.56.0
prod/1.56.0

------------------------------------------------------------------------

# 5. Стадии релиза

Важно: стадии релиза — это НЕ CI/CD pipeline. Это внутренняя кухня продукта.

Develop

ветка: develop/<version>

Значение:
- задачи релиза переданы в разработку
- разработка выполняется
- merge request вливаются в develop

Alpha

ветка: alpha/<version>

Значение:
- разработка завершена
- релиз передан в отдел тестирования
- QA проводит тестирование

Beta

ветка:  beta/<version>

Значение:
- внутреннее тестирование завершено
- релиз выкатили на бета окружение

Production

ветка: prod/<version>

Финальная выкладка релиза в production.

------------------------------------------------------------------------

# 6. Release Flow

Движение изменений происходит по цепочке:

develop -> alpha -> beta -> production

------------------------------------------------------------------------

# 7. Что контролирует система

Система должна отвечать на ключевые вопросы:
- Состав релиза
- Какие задачи входят в релиз.

Источники:
- commit messages
- merge commits
- номера задач вида TBLT-XXXX

Потерянные изменения

Например:
есть коммит в beta/1.53.0 но его нет в develop/1.54.0
Это означает, что изменение потерялось между релизами.

Полнота релиза

Проверки:
-   все ли коммиты develop попали в alpha
-   все ли коммиты alpha попали в beta

------------------------------------------------------------------------

# 8. Источники данных

Основной источник данных — GitLab API.

Он используется для получения:
-   коммитов
-   merge requests
-   веток
-   diff между ветками

------------------------------------------------------------------------

# 9. Особенности merge-коммитов

Merge commit часто выглядит так:

Merge branch ‘feature/TBLT-5933’

или

Merge branch ‘feature/TBLT-5933’ into ‘develop/1.14.0’

Иногда номер задачи присутствует только в merge commit message.

------------------------------------------------------------------------

# 10. Прогресс релиза

Прогресс релиза рассчитывается на основе:
-   количества задач
-   статусов задач
-   факта попадания изменений в ветки

Текущая реализация:
-   прогресс рассчитывается при старте экрана с релизами для всех релизов в статусе "In Progress"
-   значение сохраняется в базе данных

# 19. Сущность: Bundle (Пакет релизов)

**Bundle** — кросс-проектная группировка релизов для синхронной доставки.

**Концепция:**
- Один пакет объединяет релизы из нескольких проектов, которые должны выйти вместе
- Пример: "Tablet Delivery March 2026" включает релизы из 3 разных GitLab-проектов
- Минимум 2 проекта в одном пакете; каждый проект входит в пакет не более одного раза

**Статусы пакета:**
| Значение | Описание |
|----------|----------|
| `planned` | Планируемый |
| `in_progress` | В работе |
| `released` | Выпущен |
| `cancelled` | Отменён |

**Структура:**
- `Bundle` — сам пакет (название, статус, плановая и фактическая даты релиза)
- `BundleItem` — элемент пакета: связь конкретного проекта с конкретным релизом

**Отображение:**
- Страница `/bundles` — список пакетов в виде карточек
- Каждая карточка показывает состав (проект → версия релиза) и прогресс каждого релиза

**API пакетов:**
| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/api/bundles/create-with-items` | Создание пакета с элементами |
| GET | `/api/bundles/{id}/edit` | Форма редактирования (HTMX) |
| POST | `/api/bundles/{id}/delete` | Удаление пакета |
| POST | `/api/bundles/items/{id}/update` | Обновление релиза в элементе |
| POST | `/api/bundles/items/{id}/delete` | Удаление элемента из пакета |
| GET | `/api/bundles/new-projects/available` | Доступные проекты для нового пакета |
| GET | `/api/bundles/{id}/projects/available` | Доступные проекты для существующего пакета |
| GET | `/api/projects/{gitlab_id}/releases/in-progress` | Активные релизы проекта |

**Файлы:**
- `templates/bundles.html` — страница со списком и модальными окнами создания/редактирования
- `templates/partials/bundles_table.html` — partial для рендера списка пакетов (HTMX-swap)

# 20. Модель данных (Схема БД)

**projects** — проекты GitLab
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Primary key |
| name | TEXT | Название проекта |
| url | TEXT | URL проекта в GitLab |
| gitlab_project_id | TEXT | ID проекта в GitLab API |

**releases** — релизы проектов
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Primary key |
| project_id | INTEGER | Foreign key → projects.id |
| version | TEXT | Версия релиза (например, 1.56.0) |
| status | TEXT | Статус релиза |
| stage | TEXT | Текущая стадия (develop/alpha/beta/prod) |
| start_date | TEXT | Дата начала |
| release_date | TEXT | Дата выхода в production |
| jira_fix_version | TEXT | Fix Version в Jira |
| progress | REAL | Прогресс выполнения (0-100) |

**stages** — стадии проекта
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Primary key |
| project_id | INTEGER | Foreign key → projects.id |
| name | TEXT | Название стадии |
| stage_order | INTEGER | Порядок стадии |

**release_stage_history** — история изменений стадий
| Поле | Тип | Описание |
|------|-----|----------|
| id | INTEGER | Primary key |
| release_id | INTEGER | Foreign key → releases.id |
| old_stage | TEXT | Предыдущая стадия |
| new_stage | TEXT | Новая стадия |
| changed_at | TEXT | Дата изменения |

# 24. Типовые GitLab API запросы

**Получение коммитов между ветками:**
```bash
GET /projects/:id/repository/compare?from=<ref>&to=<ref>
```

**Получение списка веток:**
```bash
GET /projects/:id/repository/branches
```

**Получение Merge Requests:**
```bash
GET /projects/:id/merge_requests
```

**Получение diff MR:**
```bash
GET /projects/:id/merge_requests/:merge_request_iid/changes
```

# 26. Ключевые бизнес-процессы для автоматизации

**Проверка полноты релиза:**
- Все ли коммиты `develop` попали в `alpha`
- Все ли коммиты `alpha` попали в `beta`
- Все ли коммиты `beta` попали в `prod`

**Поиск потерянных изменений:**
- Коммит есть в `beta/1.53.0`, но отсутствует в `develop/1.54.0`
- Задача TBLT-XXXX есть в MR, но нет в целевой ветке

**Расчёт прогресса релиза:**
- На основе статусов задач в Jira
- На основе количества задач в каждой стадии
- Сохранение результата в БД
