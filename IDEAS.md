### [2026-03-13] 🚀 Release Health Score (Самая мощная фича!)

**Концепция:** Вместо простого прогресса показывать комплексный показатель "здоровья" релиза.

**Пример отображения:**
```
Release Health: 82%

Факторы:
├─ Progress:      67%  (вес 40%)
├─ Missing Commits: OK  (вес 25%)
├─ Open Bugs:     3    (вес 20%)
└─ Failed Pipelines: 0  (вес 15%)
```

**Формула:**
```
health_score = (
    progress * 0.40 +
    missing_commits_score * 0.25 +
    open_bugs_score * 0.20 +
    pipelines_score * 0.15
)
```

**Где:**
- `progress` — текущий прогресс выполнения (0-100%)
- `missing_commits_score` — 100% если нет отсутствующих коммитов, иначе меньше
- `open_bugs_score` — 100% если нет открытых багов, иначе уменьшается с каждым багом
- `pipelines_score` — 100% если нет упавших пайплайнов

**Преимущества:**
- Одна цифра даёт полную картину
- Можно сортировать релизы по "проблемности"
- Наглядно показывает какие релизы требуют внимания
- Можно добавить тренд (улучшается/ухудшается)

**Где показывать:**
- В таблице релизов (отдельный столбец с цветным индикатором)
- На главной странице (топ "проблемных" релизов)
- В уведомлениях (если health score упал ниже порога)

**Мысль:** Это может стать "киллер-фичей" продукта — ни одна панель не показывает здоровье релиза в одном числе!

### [2026-03-13] ⚡ Оптимизация: Один JQL запрос для всех релизов `[ ]`

**Проблема:** Сейчас для каждого релиза делается отдельный запрос к Jira. Если 10 активных релизов — 10 запросов.

**Решение:** Один JQL запрос для всех активных релизов сразу.

**Пример JQL:**
```jql
fixVersion in ("1.54.0", "1.53.0", "1.52.2") 
AND project = "TBLT" 
AND issuetype in ("Task", "Bug")
```

**API запрос:**
```bash
POST /rest/api/2/search
{
  "jql": "fixVersion in (...)",
  "fields": ["status", "fixVersions", "summary"]
}
```

**Алгоритм:**
1. Собрать все `jira_fix_version` для активных релизов
2. Сделать ОДИН запрос к Jira
3. Сгруппировать результаты по `fixVersion`
4. Обновить прогресс для каждого релиза

**Выгода:**
- 10 релизов → 1 запрос вместо 10 (в 10 раз быстрее!)
- Меньше нагрузка на Jira
- Быстрее загрузка страницы

**Код (псевдо):**
```python
versions = [r.jira_fix_version for r in active_releases]
jql = f'fixVersion in ({",".join(f\'"{v}\' for v in versions)})'
issues = jira.search(jql)

# Группируем по fixVersion
by_version = defaultdict(list)
for issue in issues:
    for fv in issue.fields.fixVersions:
        by_version[fv.name].append(issue)

# Считаем прогресс для каждого релиза
for version, issues in by_version.items():
    progress = calculate(issues)
    update_db(version, progress)
```

**Мысль:** Это критичная оптимизация для продакшена. Если будет 50+ релизов, текущий подход умрёт.

### [2026-03-13] ⏱️ Time in Stage — Время в стадиях

**Концепция:** Автоматически отслеживать сколько времени релиз провёл в каждой стадии.

**Пример таблицы:**

| Stage | Entered | Left | Duration |
|-------|---------|------|----------|
| develop | 10 Mar 11:22 | 12 Mar 09:15 | 1d 21h |
| alpha | 12 Mar 09:15 | 13 Mar 02:11 | 16h |
| beta | 13 Mar 02:11 | 13 Mar 15:32 | 13h |

**Как собирать данные:**
- Использовать существующую таблицу `release_stage_history`
- При изменении стадии записывать timestamp
- Вычислять разницу между соседними записями

**Формула:**
```python
history = get_stage_history(release_id)
for i, record in enumerate(history):
    entered = record.changed_at
    left = history[i+1].changed_at if i+1 < len(history) else now()
    duration = left - entered
```

**Где показывать:**
- Отдельная страница "/releases/{id}/time-in-stage"
- В модалке редактирования релиза (расширяемый блок)
- В истории стадий (дополнительная колонка "Duration")

**Преимущества:**
- Выявление "узких мест" в процессе релиза
- Метрики для улучшения CI/CD
- Прогнозирование будущих релизов ("обычно стадия занимает 2 дня")
- SLA мониторинг (если стадия длится дольше нормы)

**Идеи для развития:**
- 🔴 Подсветка стадий, которые длятся дольше среднего
- 📊 График "Среднее время в стадиях за последние 10 релизов"
- ⏰ Алерт если релиз "застрял" в стадии (> 3 дней)
- 📈 Экспорт метрик для аналитики

**Мысль:** Это даст ответ на вопрос "почему релиз идёт так долго?" и поможет оптимизировать процесс.

### [2026-03-13] 📏 Lead Time релиза

**Концепция:** Измерять общее время от начала разработки до попадания в прод.

**Формула:**
```
lead_time = prod_timestamp - develop_start_timestamp
```

**Пример отображения:**
```
Release Lead Time: 4 days 12 hours
```

**Как собирать данные:**
- `develop_start` — первая запись в `release_stage_history` со стадией "develop"
- `prod_timestamp` — запись со стадией "prod" или "released"
- Разница в днях/часах

**Где показывать:**
- В таблице релизов (отдельный столбец)
- На карточке релиза
- Сводная статистика на главной

**Связанные метрики:**

### Release Velocity
```
velocity = 1 / avg(lead_time)
```
- Сколько релизов в месяц/неделю
- Пример: "2.3 релиза в неделю"

### Lead Time Trend
- Увеличивается или уменьшается время релиза
- График "Lead Time по релизам"

**Преимущества:**
- Ответ на вопрос "как быстро мы релизим?"
- Возможность ставить цели ("сократить lead time до 3 дней")
- Сравнение команд/проектов
- Прогнозирование дат будущих релизов

**Идеи для развития:**
- 🎯 Целевой lead time (SLA)
- 📊 Сравнение с предыдущими релизами
- 🔴 Алерт если lead time превышает норму
- 📈 Разбивка по типам релизов (major/minor/patch)

**Мысль:** Lead Time — одна из ключевых метрик DORA (DevOps Research & Assessment). Измерение — первый шаг к улучшению!

### [2026-03-13] 🚦 Stage Bottleneck Detection

**Концепция:** Автоматически определять "узкие места" в процессе релиза на основе статистики.

**Как работает:**

1. Собираем статистику по всем завершённым релизам:

| Stage | Avg Time | Median | Max | Min |
|-------|----------|--------|-----|-----|
| develop | 3d | 2.5d | 7d | 1d |
| alpha | 8h | 6h | 24h | 2h |
| beta | 2d | 1.5d | 5d | 8h |
| prod | — | — | — | — |

2. Вычисляем bottleneck:
```python
bottleneck = max(stages, key=lambda s: s.avg_time)
```

3. Показываем результат:
```
🚦 Main Bottleneck: beta (2 days average)
   This stage takes 67% of total lead time!
```

**Где показывать:**
- Отдельная страница "/analytics/bottlenecks"
- Виджет на главной странице
- В отчёте по проекту

**Визуализация:**
```
Stage Duration Heatmap:

develop  ████████████████████ 3d
alpha    ████ 8h
beta     ██████████████ 2d     ← BOTTLENECK
prod     —
```

**Дополнительные инсайты:**
- 🔴 "beta занимает 2x больше чем в прошлом месяце"
- 📈 "trend: bottleneck смещается от develop к beta"
- ⚠️ "alpha стабильно быстрая, beta вариируется сильно"

**Преимущества:**
- Наглядное указание где оптимизировать процесс
- Ответ на вопрос "почему релизы идут медленно?"
- Data-driven решения для улучшения CI/CD
- Возможность измерить эффект от оптимизаций

**Идеи для развития:**
- 📊 Heatmap по дням недели (когда чаще релизимся)
- 🔔 Алерт если bottleneck превысил порог
- 📈 Сравнение с предыдущим периодом
- 🎯 Рекомендации ("автоматизируйте beta тестирование")

**Мысль:** Это превращает сырые данные в actionable insights — сразу видно куда прикладывать усилия для ускорения!

### [2026-03-13] 📊 Анализ времени между стадиями — Эффективность команды

**Концепция:** Отображать не время в стадиях, а время **между стадиями** как этапы жизненного цикла релиза.

**Как работает:**

Из истории стадий автоматически строится таблица переходов:

| Этап | Формула | Значение |
|------|---------|----------|
| Develop → Alpha | `alpha_timestamp - develop_timestamp` | Время разработки |
| Alpha → Beta | `beta_timestamp - alpha_timestamp` | Время тестирования |
| Beta → Prod | `prod_timestamp - beta_timestamp` | Время пилота |

**Пример таблицы по релизам:**

| Release | Dev | QA | Pilot | Total |
|---------|-----|----|-------|-------|
| 1.52 | 5d | 2d | 3d | 10d |
| 1.53 | 4d | 2d | 1d | 7d |
| 1.54 | 7d | 5d | 6d | 18d |
| 1.55 | 6d | 3d | 2d | 11d |
| **Avg** | **5.5d** | **3d** | **3d** | **11.5d** |

**Визуализация:**

```
Stacked Bar Chart — Release Timeline:

1.52  ████████ Dev (5d) ████ QA (2d) ██████ Pilot (3d)
1.53  ███████ Dev (4d)  ████ QA (2d)  ██ Pilot (1d)
1.54  ████████████ Dev (7d) █████████ QA (5d) ████████████ Pilot (6d) ← PROBLEM!
1.55  ██████████ Dev (6d) █████ QA (3d) ████ Pilot (2d)
```

**Что это показывает:**

| Проблема | Как выглядит | Действие |
|----------|--------------|----------|
| Долгая разработка | Dev колонка растёт | Упростить фичи, добавить разработчиков |
| Долгое тестирование | QA колонка растёт | Автоматизировать тесты |
| Долгий пилот | Pilot колонка растёт | Упростить процесс acceptance |
| Нестабильность | Сильные колебания между релизами | Стандартизировать процесс |

**Где показывать:**
- Страница "/analytics/team-efficiency"
- Сводка на главной ("Средний цикл релиза: 11.5 дней")
- Сравнение команд (если несколько проектов)

**Дополнительные метрики:**

#### Consistency Score
```
consistency = 1 / std_dev(total_time)
```
- Насколько предсказуем процесс
- Пример: "85% — стабильные релизы"

#### Phase Efficiency
```
dev_efficiency = avg(dev_time) / avg(total_time)
```
- Какой процент времени занимает каждая фаза
- Пример: "Dev: 48%, QA: 26%, Pilot: 26%"

**Идеи для развития:**
- 🎯 Целевые значения для каждой фазы (SLA)
- 🔔 Алерт если фаза превысила норму
- 📈 Тренд по версиям ("QA ускоряется!")
- 🏆 "Самый быстрый релиз месяца"
- 📊 Сравнение с другими командами/проектами

**Мысль:** Это превращает историю релизов в инструмент управления командой — сразу видно где процесс ломается и куда направлять усилия!

### [2026-03-13] ⚠️ Анализ загрузки тестирования — Stage Concurrency

**Концепция:** Показывать когда несколько релизов одновременно находятся в одной стадии — это создаёт нагрузку на команду.

**Проблема:**

```
Release 1.52  ████████ Alpha ████████
Release 1.53      ████████ Alpha ████████
Release 1.54          ████████ Alpha ████████
                      ↑
                   QA перегружен!
                   3 релиза одновременно
```

**Как работает:**

1. Собираем timeline всех активных релизов
2. Для каждой даты считаем сколько релизов в каждой стадии
3. Показываем "тепловую карту" загрузки

**Пример отображения:**

```
Stage Concurrency Heatmap — Alpha

        │    │    │    │    │    │    │
Mar 10  │ █  │    │    │    │    │    │  1 релиз
Mar 11  │ ██ │    │    │    │    │    │  2 релиза
Mar 12  │███ │    │    │    │    │    │  3 релиза ← WARNING!
Mar 13  │ ██ │    │    │    │    │    │  2 релиза
Mar 14  │ █  │    │    │    │    │    │  1 релиз
        │    │    │    │    │    │    │
        1.52   1.53   1.54
```

**Сигналы проблем:**

| Паттерн | Значение | Действие |
|---------|----------|----------|
| 3+ релиза в одной стадии | Критическая нагрузка | Нанять ещё QA / остановить новые релизы |
| 2 релиза долго | Средняя нагрузка | Автоматизировать тесты |
| Частые пересечения | Процесс не настроен | Ввести релизные окна |

**Где показывать:**
- Виджет на главной ("⚠️ QA загружен: 3 релиза в Alpha")
- Страница "/analytics/workload"
- Календарь релизов с подсветкой перегрузок

**Метрики:**

#### Max Concurrency
```
max_concurrency = max(releases_in_stage)
```
- Пиковая нагрузка на стадию
- Пример: "QA: 3 релиза одновременно"

#### Overload Days
```
overload_days = count(days_with_3+_releases)
```
- Сколько дней было перегрузок
- Пример: "5 дней за месяц с 3+ релизами"

#### Stage Utilization
```
qa_utilization = total_qa_days / (calendar_days * qa_team_size)
```
- Процент использования QA команды
- Пример: "QA загрузка: 140% — нужна помощь!"
