## [2026-03-13] 🚀 Release Health Score (Самая мощная фича!)

**Концепция:** Вместо простого прогресса показывать комплексный показатель "здоровья" релиза.

**Пример отображения:**
```
Release Health: 82%

Факторы:
├─ Progress:      67%  (вес 40%)
├─ Missing Commits: OK  (вес 25%)
├─ Open Bugs:     3    (вес 20%)
└─ Failed Pipelines: 0  (вес 15%)
```

**Формула:**
```
health_score = (
    progress * 0.40 +
    missing_commits_score * 0.25 +
    open_bugs_score * 0.20 +
    pipelines_score * 0.15
)
```

**Где:**
- `progress` — текущий прогресс выполнения (0-100%)
- `missing_commits_score` — 100% если нет отсутствующих коммитов, иначе меньше
- `open_bugs_score` — 100% если нет открытых багов, иначе уменьшается с каждым багом
- `pipelines_score` — 100% если нет упавших пайплайнов

**Преимущества:**
- Одна цифра даёт полную картину
- Можно сортировать релизы по "проблемности"
- Наглядно показывает какие релизы требуют внимания
- Можно добавить тренд (улучшается/ухудшается)

**Где показывать:**
- В таблице релизов (отдельный столбец с цветным индикатором)
- На главной странице (топ "проблемных" релизов)
- В уведомлениях (если health score упал ниже порога)

**Мысль:** Это может стать "киллер-фичей" продукта — ни одна панель не показывает здоровье релиза в одном числе!

## [2026-03-13] ⚡ Оптимизация: Один JQL запрос для всех релизов

**Проблема:** Сейчас для каждого релиза делается отдельный запрос к Jira. Если 10 активных релизов — 10 запросов.

**Решение:** Один JQL запрос для всех активных релизов сразу.

**Пример JQL:**
```jql
fixVersion in ("1.54.0", "1.53.0", "1.52.2") 
AND project = "TBLT" 
AND issuetype in ("Task", "Bug")
```

**API запрос:**
```bash
POST /rest/api/2/search
{
  "jql": "fixVersion in (...)",
  "fields": ["status", "fixVersions", "summary"]
}
```

**Алгоритм:**
1. Собрать все `jira_fix_version` для активных релизов
2. Сделать ОДИН запрос к Jira
3. Сгруппировать результаты по `fixVersion`
4. Обновить прогресс для каждого релиза

**Выгода:**
- 10 релизов → 1 запрос вместо 10 (в 10 раз быстрее!)
- Меньше нагрузка на Jira
- Быстрее загрузка страницы

**Код (псевдо):**
```python
versions = [r.jira_fix_version for r in active_releases]
jql = f'fixVersion in ({",".join(f\'"{v}\' for v in versions)})'
issues = jira.search(jql)

# Группируем по fixVersion
by_version = defaultdict(list)
for issue in issues:
    for fv in issue.fields.fixVersions:
        by_version[fv.name].append(issue)

# Считаем прогресс для каждого релиза
for version, issues in by_version.items():
    progress = calculate(issues)
    update_db(version, progress)
```

**Мысль:** Это критичная оптимизация для продакшена. Если будет 50+ релизов, текущий подход умрёт.

## [2026-03-13] ⏱️ Time in Stage — Время в стадиях

**Концепция:** Автоматически отслеживать сколько времени релиз провёл в каждой стадии.

**Пример таблицы:**

| Stage | Entered | Left | Duration |
|-------|---------|------|----------|
| develop | 10 Mar 11:22 | 12 Mar 09:15 | 1d 21h |
| alpha | 12 Mar 09:15 | 13 Mar 02:11 | 16h |
| beta | 13 Mar 02:11 | 13 Mar 15:32 | 13h |

**Как собирать данные:**
- Использовать существующую таблицу `release_stage_history`
- При изменении стадии записывать timestamp
- Вычислять разницу между соседними записями

**Формула:**
```python
history = get_stage_history(release_id)
for i, record in enumerate(history):
    entered = record.changed_at
    left = history[i+1].changed_at if i+1 < len(history) else now()
    duration = left - entered
```

**Где показывать:**
- Отдельная страница "/releases/{id}/time-in-stage"
- В модалке редактирования релиза (расширяемый блок)
- В истории стадий (дополнительная колонка "Duration")

**Преимущества:**
- Выявление "узких мест" в процессе релиза
- Метрики для улучшения CI/CD
- Прогнозирование будущих релизов ("обычно стадия занимает 2 дня")
- SLA мониторинг (если стадия длится дольше нормы)

**Идеи для развития:**
- 🔴 Подсветка стадий, которые длятся дольше среднего
- 📊 График "Среднее время в стадиях за последние 10 релизов"
- ⏰ Алерт если релиз "застрял" в стадии (> 3 дней)
- 📈 Экспорт метрик для аналитики

**Мысль:** Это даст ответ на вопрос "почему релиз идёт так долго?" и поможет оптимизировать процесс.
